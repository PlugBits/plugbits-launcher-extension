# 次期アップデート候補: Excel Overlay のビュー列連動（表示列をkintoneビューに合わせる）

status: 実装可能・未着手（v3.0.1時点の調査済みメモ）

## 何をやるか

kintoneのビューで設定した表示列を、Excel Overlay の初期表示列として反映する。
現状は常に「アプリの全フィールド（編集対応タイプ・フォーム順）＋レイアウトプリセット」で、
ビューを切り替えても列構成が変わらない。

## 過去に断念した理由と、解消済みであること

- 断念理由: 「表示中のビューを確実に特定できない」こと。アプリトップ(?view=なし)では
  kintone.app.getViewName() が undefined を返し（実アプリで確認済み）、当時の実装は
  ビュー一覧のキー順先頭に誤フォールバックしていた（プロセス管理アプリでは組み込みの
  「（作業者が自分）」を掴む）。間違ったビューの列に絞るのは全列表示より有害。
- 解消: v3.0.1 のビュー特定修正（content-overlay-controller.js の pickViewFromViews /
  resolveViewInfoFromMetadata）で以下が保証された:
  1. ?view=ID → ID厳密一致
  2. アプリトップ → index最小ビューを候補にし、ページクエリ(kintone.app.getQuery())の
     条件部と候補の filterCond が一致する場合のみ採用
  3. 不一致 → ページクエリをそのまま使う安全フォールバック（誤絞り込みが構造的に不可能）
  回帰テスト: tests/query-split-unit.test.mjs（実アプリのビュー構成データ使用）

## データソース（確認済み）

- ビューの表示列は GET /k/v1/app/views の各ビューの `fields`（フィールドコード配列）に入る。
  コンソール確認用:
  ```js
  const r = await kintone.api(kintone.api.url('/k/v1/app/views.json', true), 'GET', { app: kintone.app.getId() });
  console.log(Object.fromEntries(Object.entries(r.views).map(([name, v]) => [name, v.fields])));
  ```
- 拡張側の配管は既存:
  - 取得: `extractViewFieldOrder(view)`（view.columns → view.fields → view.layout の順で列コード配列化）
  - 保持: overlay起動時に `this.viewFieldOrder` へ格納済み
    （現状はフィールドメタ取得失敗時の非常用フォールバックでのみ使用）
  - プリセットはビュー単位のキー（`computeViewKey`: id:/name:/param:/default）で保存されている

## 推奨仕様（設計判断のメモ）

- **プリセット未保存のビューでは、ビューの列＋列順を初期表示にする。
  プリセットを保存したらそちらが優先**（既存ユーザーの保存済みレイアウトを壊さない。
  初見のビューは一覧と同じ見た目で開く＝驚き最小）
- ビュー列にオーバーレイ非対応フィールド（関連レコード・STATUS等 allowedTypes外）が
  含まれる場合は除外
- ビュー特定に失敗した安全フォールバック時（all_records扱い）は従来どおり全列
- 列順ダイアログ（表示列チェックボックス）とレイアウトプリセットの既存挙動は不変

## 実装の当たり

- `applyListFieldScope` 系: プリセット未保存（activePresetが初期状態）かつ
  `viewFieldOrder` 非空のとき、初期 visibleColumns/columnOrder として
  `viewFieldOrder`（allowedTypesでフィルタ）を流し込む
- 規模: 小（データ・特定ロジック・プリセット機構すべて既存。差分は初期値の決定のみ）
- テスト: query-split-unit の resolveViewInfo スタブパターン＋overlay E2E
  （tests/helpers/overlay-page.mjs）が流用可能
